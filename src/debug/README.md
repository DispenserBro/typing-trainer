# Debug Module

Этот модуль содержит локальные инструменты для диагностики, отладки и технической проверки приложения.

Он не участвует в обычном рантайме `Typing Trainer` и нужен только для разработки.

## Зачем нужен модуль

`src/debug` нужен для задач, которые неудобно или опасно решать прямо в UI:

- массовая проверка генерации практики
- поиск перекосов в текстах и словарях
- отладка адаптивной логики
- проверка миграций и сохранений
- диагностические отчеты по игровому балансу

Идея простая: все такие технические инструменты должны жить в одном месте, а не расползаться по `pages`, `contexts` и `engine`.

## Как запускать

### Базовый прогон

```bash
npm run diagnostics:practice
```

Что делает:

- собирает debug-инструменты через `tsconfig.debug.json`
- запускает анализ генерации практики
- по умолчанию проверяет `qwerty` и `йцукен`
- по умолчанию проверяет оба режима: `normal` и `rhythm`

### Проверка с большим количеством прогонов

```bash
npm run diagnostics:practice -- --runs 400
```

Полезно, если нужно увидеть более стабильное распределение букв, биграмм и повторов.

### Проверка конкретной раскладки

```bash
npm run diagnostics:practice -- --layout qwerty
```

```bash
npm run diagnostics:practice -- --layout йцукен
```

### Проверка только одного режима

```bash
npm run diagnostics:practice -- --mode rhythm
```

```bash
npm run diagnostics:practice -- --mode normal
```

### Проверка поведения умной адаптации

```bash
npm run diagnostics:practice -- --focus chars --strength high
```

```bash
npm run diagnostics:practice -- --focus bigrams --strength medium
```

```bash
npm run diagnostics:practice -- --focus rhythm --mode rhythm
```

### Проверка без адаптации

```bash
npm run diagnostics:practice -- --adaptation off
```

Это полезно, когда нужно сравнить обычную генерацию и адаптивную.

### Проверка частично открытого набора букв

```bash
npm run diagnostics:practice -- --layout йцукен --unlock 12
```

Это помогает понять, как генератор ведет себя на ранних этапах практики.

### Сохранение отчета в JSON

```bash
npm run diagnostics:practice -- --runs 300 --json reports/practice-diagnostics.json
```

Это удобно, если отчет нужно:

- сохранить для сравнения
- приложить к задаче
- потом разобрать программно

## Поддерживаемые аргументы CLI

- `--layout <id>`
  Конкретная раскладка, например `qwerty` или `йцукен`.
- `--runs <number>`
  Сколько текстов сгенерировать для одного сценария.
- `--unlock <number>`
  Сколько букв считать открытыми.
- `--mode <normal|rhythm>`
  Проверять только один режим практики.
- `--adaptation <on|off>`
  Включить или выключить умную адаптацию.
- `--strength <low|medium|high>`
  Сила адаптации.
- `--focus <balanced|chars|bigrams|rhythm>`
  Главный акцент адаптации.
- `--json <path>`
  Путь для сохранения JSON-отчета.

## Что именно проверяет диагностика практики

Инструмент считает:

- частоты букв
- частоты биграмм
- самые частые слова
- среднюю длину слов
- долю уникальных слов
- долю повторов
- долю слов вне словаря

Дополнительно он поднимает `warnings`, если замечает:

- слишком сильный перекос по одной букве
- слишком сильный перекос по одной биграмме
- низкое разнообразие слов
- слишком много повторов
- слишком короткие слова
- слишком много синтетических слов

## Как читать отчет

Пример секции:

```text
=== qwerty / normal ===
Layout: qwerty (en)
Mode: normal | Runs: 80 | Unlocked: 26
Smart adaptation: on | strength=medium | focus=balanced
Words/text: 16 | Avg word length: 3.78
Unique ratio: 86.3% | Non-dictionary: 1.9% | Repeats: 13.7%
Top chars: a:10.14%, m:9.75%, e:9.48%
Warnings:
- Явных перекосов не найдено по выбранным эвристикам.
```

На что смотреть в первую очередь:

- `Top chars`
  Не доминирует ли одна буква слишком сильно.
- `Top bigrams`
  Нет ли навязчивого повтора одной пары символов.
- `Unique ratio`
  Если слишком низкий, тексты однообразные.
- `Repeats`
  Если слишком высокий, генератор слишком часто повторяет слова.
- `Non-dictionary`
  Если слишком высокий в обычном режиме, текст может ощущаться неестественным.
- `Warnings`
  Краткое итоговое резюме по эвристикам.

## Как добавлять новые debug-инструменты

Рекомендуемый шаблон:

1. Создать отдельный модуль в `src/debug/`, например:
   - `gameBalanceDiagnostics.ts`
   - `saveMigrationDiagnostics.ts`
   - `layoutDiagnostics.ts`
2. Вынести в него:
   - типы входных параметров
   - типы отчета
   - чистую функцию запуска диагностики
   - форматтер вывода
3. Создать отдельный CLI entrypoint:
   - `runGameBalanceDiagnostics.ts`
   - `runSaveMigrationDiagnostics.ts`
4. Добавить npm-команду в [package.json](/d:/Danil/Projects/TypingTrainer/package.json)
5. Если инструмент полезен долгосрочно, дописать его в этот `README`

## Рекомендации по проектированию debug-инструментов

- Не тянуть в debug-модуль React-компоненты и UI.
- Предпочитать чистые функции и явные входные параметры.
- Держать формат отчета стабильным, если он может использоваться повторно.
- Если нужен файл-отчет, поддерживать JSON наряду с текстовым выводом.
- Не смешивать диагностику с боевой логикой приложения.
- Если логика может пригодиться в нескольких инструментах, выносить ее в отдельный helper внутри `src/debug`.

## Ограничения текущей реализации

- Диагностика генерации практики пока опирается на эвристики, а не на полноценные snapshot-тесты.
- Для адаптивной практики сейчас используются синтетические слабые места, а не реальные сохранения пользователя.
- Отчет оценивает качество текста статистически, но не “понимает” его как человек.

Это нормально для первой версии debug-модуля: его задача быстро ловить технические перекосы, а не заменять ручной UX-review.
