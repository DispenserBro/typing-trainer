# Дорожная карта

Этот файл описывает рабочий план развития `Typing Trainer` и правила сопровождения проекта. Он нужен не для хранения всех идей подряд, а для удержания ближайшего фокуса: что исправлять в приложении, как проверять изменения и как поддерживать документацию в актуальном состоянии.

## Условия работы с проектом

- Основной продуктовый код находится в `src/`; собранные каталоги `dist`, `dist-debug`, `dist-build` и `build` не должны становиться источником правды.
- Изменения в поведении приложения сначала вносятся в исходники и проверяются диагностикой, а только затем отражаются в документации.
- Пользовательские сценарии важнее внутренних названий модулей: если меняется логика первого запуска, режимов, статистики, расширений или игры, нужно проверить соответствующий пользовательский путь целиком.
- В активной работе должно быть не больше 3 направлений одновременно.
- В roadmap остаются только незакрытые задачи, ближайшие технические цели и правила сопровождения, которые помогают выполнять эти задачи.
- Выполненные этапы после проверки удаляются из roadmap, чтобы файл оставался списком актуальной работы.
- Если задача не влияет на ближайший цикл, она не должна засорять активный фокус.
- Нельзя переписывать основные проблемные блоки roadmap только ради стилистики: активные направления должны оставаться видимыми, пока не закрыты фактически.

## Правила документации

- Проектная документация по умолчанию пишется на русском языке.
- Пользовательские инструкции должны объяснять сценарии использования приложения, а не внутреннее устройство кода.
- Документация для разработчиков должна фиксировать архитектурные решения, контракты, ограничения и диагностические команды, которые реально помогают сопровождать проект.
- При изменениях в расширениях, аддонах, модах, темах или Mod API нужно синхронизировать `docs/`, `wiki/` и README/roadmap-материалы, если они описывают затронутый контракт.
- Если поведение приложения изменилось, документация должна описывать новое поведение после проверки, а не предполагаемое намерение.
- Документы не должны дублировать длинные списки задач из roadmap: roadmap отвечает за фокус разработки, а `docs/` и `wiki/` отвечают за устойчивые инструкции и справочные материалы.
- Перед релизом нужно сверять, что README, пользовательские wiki-страницы и developer docs не противоречат текущим сборочным сценариям, форматам расширений и диагностическим командам.

Активных незакрытых задач, которые можно закрыть внутри репозитория без доступа к внешним GitHub-секретам, сертификатам и отдельному SDK-репозиторию, сейчас нет.

## Критерии готовности изменений

- Для любого TypeScript-изменения должен проходить `npm run build`.
- Для изменений core/helper/view-model слоёв должен проходить `npm run diagnostics:quick`.
- Для изменений генерации текста должен дополнительно проходить `npm run diagnostics:content-pipeline`.
- Для изменений истории, статистики и result-flow должен дополнительно проходить `npm run diagnostics:history`.
- Для изменений центра расширений должен дополнительно проходить `npm run diagnostics:extensions`.
- Для изменений локализации должен дополнительно проходить `npm run diagnostics:i18n`.
- Для изменений игрового баланса должен дополнительно проходить `npm run diagnostics:game-balance`.
- Для релизных изменений должен проходить `npm run diagnostics:release-hardening` и соответствующий SDK gate, если затронуты публичные контракты расширений.
