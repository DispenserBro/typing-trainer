# Правила производительности renderer-слоя

Документ описывает технические правила для React renderer-слоя. Цель - уменьшить лишние пересчёты, сохранить предсказуемость состояния и отделить доменную логику от компонентов.

## Часто меняющееся состояние

Часто обновляемое состояние не должно находиться в широком `AppContext`, если оно не требуется большинству экранов.

Примеры write-hot состояния:

- активный символ;
- позиция ввода;
- preview-флаги;
- transient UI-флаги;
- индикаторы live-сессии.

Такое состояние следует держать в локальном hook-state, узком контексте или специализированном hook.

## Derived-данные

Сложные derived-данные должны строиться вне JSX:

- summary;
- рекомендации;
- карточки рекордов;
- history-агрегации;
- comparison;
- view-model для result-flow.

Если экран использует `filter`, `reduce`, `sort` или несколько проходов по `history`, вычисление следует рассматривать как кандидата на перенос в `src/core`.

## Обновления progress

Обновления `progress` должны проходить через единый commit-path, который:

- возвращает `prev` при no-op;
- синхронизирует актуальный ref состояния;
- выполняет один persist-path на логическое изменение.

Не следует добавлять локальные варианты `setProgress` и `saveProgress` с разной семантикой.

## Immutable-обновления

Нельзя мутировать вложенные структуры из предыдущего состояния.

Для `history`, `keyStats`, `practiceInsights`, `practiceRhythmHistory` и аналогичных веток нужно копировать только изменяемое поддерево. Это снижает риск скрытых мутаций и не пересобирает весь объект без необходимости.

## Разделение read-mostly и write-hot данных

Read-mostly данные:

- раскладки;
- словари;
- каталоги расширений;
- каталоги тем;
- локализационные ресурсы.

Write-hot данные:

- текущая сессия ввода;
- live-метрики;
- preview-state;
- transient UI-state.

При проектировании нового экрана нужно заранее определить, к какому классу относится каждое состояние.

## No-op guards

Если новое значение совпадает с текущим, callback должен завершиться до `setState` и до persist-вызова. Это особенно важно для настроек, профилей, practice settings и прогресса.

## Selectors и view-model helpers

Если один экран собирает одну и ту же view-model из `progress` больше одного раза, вычисление нужно вынести в helper или selector.

Рекомендуемое размещение:

- доменная логика - `src/core/<domain>/`;
- CLI-диагностика - `src/debug/`;
- React wiring - `src/renderer/hooks/`;
- отображение - `src/renderer/components/`.
