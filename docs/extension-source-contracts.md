# Контракты источников расширений

Документ описывает developer-facing контракты центра расширений. Он нужен авторам сторонних источников, аддонов, модификаций и тем, но не означает публикацию отдельного SDK-пакета.

## Источник расширений

Источник описывается файлом `manifest.json`.

Обязательные поля:

- `manifestVersion` - версия формата, сейчас `1`;
- `id` - стабильный идентификатор в kebab-case;
- `name` - отображаемое имя источника;
- `version` - версия источника;
- `type` - значение `extension-source`;
- `lists` - объект со ссылками на списки пакетов.

Поддерживаемые списки:

- `lists.addons` - JSON-массив идентификаторов аддонов;
- `lists.mods` - JSON-массив идентификаторов модификаций;
- `lists.themes` - JSON-массив идентификаторов тем.

Каждый элемент списка должен быть строкой. Для записи `example-addon` приложение ожидает manifest пакета по пути `example-addon/manifest.json` относительно файла списка.

## Карточки описания

Источник может указать Markdown-карточки:

- `cards.source` - описание источника;
- `cards.addons` - резервная директория или ссылка для карточек аддонов;
- `cards.mods` - резервная директория или ссылка для карточек модификаций;
- `cards.themes` - резервная директория или ссылка для карточек тем.

Для отдельного пакета приложение ищет `card.md` или `README.md` рядом с manifest пакета. Если Markdown отсутствует, установка пакета не блокируется.

## Аддоны

Аддон описывается manifest-файлом с `type: "content"`.

Ключевые поля:

- `manifestVersion`;
- `id`;
- `name`;
- `version`;
- `description`;
- `author`;
- `icon`;
- `minAppVersion`;
- `dependencies`;
- `resources`.

`dependencies` содержит идентификаторы модификаций или других пакетов, которые должны быть установлены для полного сценария работы. Для аддонов отсутствующие зависимости отображаются как предупреждение и не блокируют установку.

## Модификации

Модификация описывается manifest-файлом с `type: "mod"`.

Ключевые поля:

- `manifestVersion`;
- `id`;
- `name`;
- `version`;
- `entry`;
- `files`;
- `permissions`;
- `dependencies`;
- `minAppVersion`.

Для удалённых источников поле `files` обязательно для прямой установки. Если `files` отсутствует, запись остаётся доступной в каталоге, но получает `installSupport: "manual"` и не устанавливается напрямую.

`permissions` использует закрытый список значений Mod API. Неизвестное permission делает manifest модификации невалидным, чтобы каталог, диагностика и UI показывали только реально поддерживаемые зоны доступа.

Закрытые значения manifest type для аддонов, модификаций, тем и источников расширений, типов источников (`local`, `url`, `github`), статусов каталога, поддержки установки, duplicate-рекомендаций, issue stages/severity/fallback и sync-status хранятся в shared-слое. UI, loader, диагностика и будущие SDK-обёртки должны использовать эти константы как единый источник правды, а не заводить отдельные строковые списки.

Логика блокировки, причины блокировки, доступности установки и наличия пользовательского внимания для записи каталога также находится в shared-слое. Renderer и loader должны использовать общие helper-функции, чтобы сортировка, preflight, сообщения об ошибках и кнопки установки не расходились по смыслу.

Runtime validators Mod API также находятся в shared-слое: `isModPermission`, `normalizeModPermissions`, `isModUserSettingKey`, `isModUserSettingValue`, `isModGameItemDefinition`, `isModGameAchievementDefinition`, `isModInterfaceLocaleDefinition`, `isModPanel` и `isModModeDefinition` должны использоваться loader-ом, runtime и будущими SDK-обёртками как один контракт проверки.

Все mutating-методы Mod API проходят permission gate до изменения состояния. Если permission отсутствует, runtime пишет warning и не меняет `ModAPIState`.

Rule overrides модификаций также используют закрытый контракт Mod API. Runtime принимает только известные rule id и значения ожидаемого типа; некорректные overrides игнорируются и не попадают в итоговое состояние эффектов мода.

События Mod API используют такой же закрытый подход: подписки принимаются только для известных event names и только с функцией-обработчиком.

UI-панели и модовые режимы проходят runtime-проверку перед применением. Панели принимают только поддерживаемые locations `page-top`, `page-bottom` и `overlay`; режимы принимают только группы `top` и `bottom`.

Word и lesson эффекты модификаций нормализуются перед применением. Слова должны быть непустыми строками, а уроки должны иметь непустые `id`, `name` и массив строк `keys`.

Settings, item, achievement и i18n эффекты модификаций также валидируются до применения. Settings override принимает только известные ключи `UserSettings` и значения ожидаемого типа; игровые предметы проверяются по обязательным полям, закрытым enum-значениям и форме effects/durability rules; достижения требуют непустые `id`, `name`, `description`, `category` и корректные conditions; interface locales требуют непустые `id`, `label`, `nativeLabel` и словарь переводов.

Отсутствующие runtime-зависимости модификации блокируют установку.

## Темы

Тема описывается manifest-файлом с `type: "theme"`.

Ключевые поля:

- `manifestVersion`;
- `id`;
- `name`;
- `version`;
- `description`;
- `author`;
- `icon`;
- `minAppVersion`;
- `style`;
- `preview`.

Темы устанавливаются как обычные записи каталога и проходят общую проверку совместимости приложения.

Нормализация theme style primitives находится в shared-слое: `normalizeThemeColors`, `normalizeThemeStringRecord`, `normalizeThemeStringArray`, `normalizeThemeOptionalString` и `hasThemeStyleContent` используются loader-ом и должны переиспользоваться будущими SDK-обёртками. Цветовой набор считается валидным только при наличии обязательных `bg`, `surface`, `surface2`, `text`, `subtext`, `accent`, `green`, `red` и `yellow`; пустые строки в class/attribute/variable списках отбрасываются.

## Preflight-контракт каталога

Перед установкой приложение может проверить запись каталога через `validateExtensionCatalogEntry`.

Входные параметры:

- `sourceId` - идентификатор установленного источника;
- `kind` - `addons`, `mods` или `themes`;
- `entryId` - идентификатор записи из списка источника.

Результат проверки:

- `ok` - запись можно устанавливать напрямую;
- `blocked` - установка заблокирована;
- `error` - причина блокировки;
- `entry` - нормализованная запись каталога;
- `issues` - предупреждения и ошибки, найденные при проверке.

Проверка не устанавливает пакет и не изменяет пользовательские данные.

## Статусы записи каталога

`status` показывает текущее состояние записи:

- `available` - доступна для установки;
- `installed` - уже установлена;
- `update-available` - доступна новая версия;
- `source-disabled` - источник отключён;
- `source-error` - источник не синхронизирован или использует stale-cache;
- `incompatible` - пакет требует более новую версию приложения;
- `invalid` - manifest записи не удалось прочитать или проверить.

## Поддержка установки

`installSupport` принимает значения:

- `direct` - приложение может установить пакет из каталога;
- `manual` - пакет требует ручной установки.

`manual` не является ошибкой manifest. Это явный контракт для записей, которые можно показывать в каталоге, но нельзя безопасно установить автоматически.

## Issues

Каждая проблема описывается объектом `ExtensionCatalogIssue`.

Ключевые поля:

- `stage` - `manifest`, `list`, `card` или `package`;
- `severity` - `warning` или `error`;
- `message` - техническое описание;
- `code` - стабильный код для UI и диагностики;
- `target` - путь или URL проблемного ресурса;
- `fallback` - поведение приложения.

Поддерживаемые fallback-значения:

- `stale-cache` - используется последний валидный кэш источника;
- `skipped-card` - Markdown-карточка пропущена;
- `manual-only` - доступна только ручная установка;
- `blocked-install` - установка заблокирована.

## Совместимость

Если manifest пакета содержит `minAppVersion`, каталог сравнивает его с текущей версией приложения.

Несовместимый пакет получает:

- `compatibility.compatible: false`;
- issue с fallback `blocked-install`;
- статус `incompatible`, если пакет ещё не установлен.

Если несовместимая запись является обновлением уже установленного пакета, статус остаётся `update-available`, но preflight блокирует установку.

## Проверка источника

После изменения формата источника или manifest пакетов запускать:

```bash
npm run diagnostics:extensions
```

Диагностика проверяет локальные, URL и GitHub-like источники, Markdown-карточки, зависимости, совместимость версий, неизвестные permissions модификаций, manual-only записи, stale-cache и preflight-блокировки.
