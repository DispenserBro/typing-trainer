# Контракты источников расширений

Документ описывает публичные контракты центра расширений. Он нужен авторам сторонних источников, аддонов, модификаций и тем.

## Источник расширений

Источник описывается файлом `manifest.json`.

Обязательные поля:

- `manifestVersion` - версия формата, сейчас `1`;
- `id` - стабильный идентификатор в kebab-case;
- `name` - отображаемое имя источника;
- `version` - версия источника;
- `type` - значение `extension-source`;
- `lists` - объект со ссылками на списки пакетов.

Поддерживаемые списки:

- `lists.addons` - JSON-массив идентификаторов аддонов;
- `lists.mods` - JSON-массив идентификаторов модификаций;
- `lists.themes` - JSON-массив идентификаторов тем.

Каждый элемент списка должен быть строкой. Для записи `example-addon` приложение ожидает manifest пакета по пути `example-addon/manifest.json` относительно файла списка.

## Карточки описания

Источник может указать Markdown-карточки:

- `cards.source` - описание источника;
- `cards.addons` - резервная директория или ссылка для карточек аддонов;
- `cards.mods` - резервная директория или ссылка для карточек модификаций;
- `cards.themes` - резервная директория или ссылка для карточек тем.

Для отдельного пакета приложение ищет `card.md` или `README.md` рядом с manifest пакета. Если Markdown отсутствует, установка пакета не блокируется.

## Аддоны

Аддон описывается manifest-файлом с `type: "content"`.

Ключевые поля:

- `manifestVersion`;
- `id`;
- `name`;
- `version`;
- `description`;
- `author`;
- `icon`;
- `minAppVersion`;
- `dependencies`;
- `resources`.

`dependencies` содержит идентификаторы модификаций или других пакетов, которые должны быть установлены для полного сценария работы. Для аддонов отсутствующие зависимости отображаются как предупреждение и не блокируют установку.

## Модификации

Модификация описывается manifest-файлом с `type: "mod"`.

Ключевые поля:

- `manifestVersion`;
- `id`;
- `name`;
- `version`;
- `entry`;
- `files`;
- `permissions`;
- `dependencies`;
- `minAppVersion`.

Для удалённых источников поле `files` обязательно для прямой установки. Если `files` отсутствует, запись остаётся доступной в каталоге, но получает `installSupport: "manual"` и не устанавливается напрямую.

Отсутствующие runtime-зависимости модификации блокируют установку.

## Темы

Тема описывается manifest-файлом с `type: "theme"`.

Ключевые поля:

- `manifestVersion`;
- `id`;
- `name`;
- `version`;
- `description`;
- `author`;
- `icon`;
- `minAppVersion`;
- `style`;
- `preview`.

Темы устанавливаются как обычные записи каталога и проходят общую проверку совместимости приложения.

## Preflight-контракт каталога

Перед установкой приложение может проверить запись каталога через `validateExtensionCatalogEntry`.

Входные параметры:

- `sourceId` - идентификатор установленного источника;
- `kind` - `addons`, `mods` или `themes`;
- `entryId` - идентификатор записи из списка источника.

Результат проверки:

- `ok` - запись можно устанавливать напрямую;
- `blocked` - установка заблокирована;
- `error` - причина блокировки;
- `entry` - нормализованная запись каталога;
- `issues` - предупреждения и ошибки, найденные при проверке.

Проверка не устанавливает пакет и не изменяет пользовательские данные.

## Статусы записи каталога

`status` показывает текущее состояние записи:

- `available` - доступна для установки;
- `installed` - уже установлена;
- `update-available` - доступна новая версия;
- `source-disabled` - источник отключён;
- `source-error` - источник не синхронизирован или использует stale-cache;
- `incompatible` - пакет требует более новую версию приложения;
- `invalid` - manifest записи не удалось прочитать или проверить.

## Поддержка установки

`installSupport` принимает значения:

- `direct` - приложение может установить пакет из каталога;
- `manual` - пакет требует ручной установки.

`manual` не является ошибкой manifest. Это явный контракт для записей, которые можно показывать в каталоге, но нельзя безопасно установить автоматически.

## Issues

Каждая проблема описывается объектом `ExtensionCatalogIssue`.

Ключевые поля:

- `stage` - `manifest`, `list`, `card` или `package`;
- `severity` - `warning` или `error`;
- `message` - техническое описание;
- `code` - стабильный код для UI и диагностики;
- `target` - путь или URL проблемного ресурса;
- `fallback` - поведение приложения.

Поддерживаемые fallback-значения:

- `stale-cache` - используется последний валидный кэш источника;
- `skipped-card` - Markdown-карточка пропущена;
- `manual-only` - доступна только ручная установка;
- `blocked-install` - установка заблокирована.

## Совместимость

Если manifest пакета содержит `minAppVersion`, каталог сравнивает его с текущей версией приложения.

Несовместимый пакет получает:

- `compatibility.compatible: false`;
- issue с fallback `blocked-install`;
- статус `incompatible`, если пакет ещё не установлен.

Если несовместимая запись является обновлением уже установленного пакета, статус остаётся `update-available`, но preflight блокирует установку.

## Проверка источника

После изменения формата источника или manifest пакетов запускать:

```bash
npm run diagnostics:extensions
```

Диагностика проверяет локальные, URL и GitHub-like источники, Markdown-карточки, зависимости, совместимость версий, manual-only записи, stale-cache и preflight-блокировки.
